## Agent 核心循环重构：Planner→Executor→Reviewer → Reasoner→Tool→Observation→MemoryUpdate→Critic

### 改造目标
将旧的三节点批处理模式（Planner一次性规划→Executor批量执行→Reviewer评审）重构为更灵活的五节点迭代推理循环。

### 新架构流程
```
User Goal → Reasoner(LLM思考) → 是否需要工具?
  是 → Tool → Observation → MemoryUpdate → Critic → 继续? → Reasoner
  否 → Final Answer
```

### 变更清单

1. **新建 `agent/core/state.py`** — 新 AgentState 定义
   - 旧: messages, plan, execution_results, execution_trace, review_feedback, is_complete, revised_plan, conv_id, step_count
   - 新: goal, thought, action, observation, trajectory, hypothesis, memory_context, finished, answer, iteration, conv_id
   - 从"Workflow State"（批处理状态）变为"Agent State"（迭代推理状态）

2. **新建 `agent/core/nodes.py`** — 四个核心节点实现
   - `reasoner_node`: 推理并决定下一步 Action（调用工具 或 直接回答）
   - `tool_node`: 执行单个工具调用，返回 observation
   - `memory_update_node`: 将 thought+action+observation 追加到 trajectory，递增 iteration
   - `critic_node`: 评估轨迹是否充分解决目标，决定继续或完成
   - `should_use_tool`: Reasoner 条件边（有 tool→tool 节点，无 tool→memory_update）
   - `should_continue`: Critic 条件边（finished→END，否则→reasoner）

3. **重构 `app/services/graph_service.py`** — 主入口
   - 移除旧的 AgentState, planner_node, executor_node, reviewer_node, _should_continue
   - 移除 _build_planner_prompt, _build_reviewer_prompt
   - 导入新 AgentState 和节点
   - 图结构: reasoner → (tool|memory_update) → critic → (reasoner|END)
   - run_graph_stream: 直接初始化 goal+memory_context 进入循环，不再预调 Planner LLM
   - _handle_uploaded_files: 提取为独立函数
   - _summarize_trajectory: 替代旧的 feedback 为空时的总结逻辑
   - _extract_preferences: 适配新 trajectory 结构

4. **新建 `agent/core/__init__.py`** — 包初始化

### 验证结果

## 深化改造：节点拆分 + Observation 节点

### 改造目标
1. Reasoner 独立文件 + prompt 聚焦「根据当前状态决定下一步」（不是规划）
2. Tool Runner 独立文件（一次只执行一个 action，而非批量执行 plan）
3. 新增 Observation 节点 — Agent 与 Workflow 最大区别

### 变更清单

1. **新建 `agent/core/reasoner.py`**
   - `reason(state, client, prompt_builder)`: 推理并决定下一步 Action
   - `build_reasoner_prompt()`: prompt 聚焦「目标 + 轨迹 + 当前观察 → 决定下一步」
   - prompt 中新增「当前观察」区段（findings + next_step_suggestion）
   - 与旧 Planner 的区别: 不是一次生成完整计划，而是每次只决定一步

2. **新建 `agent/core/tool_runner.py`**
   - `execute_action(state, registry)`: 执行单个工具调用
   - 与旧 Executor 的区别: 一次只执行一个 action（从 state["action"] 读取），而非遍历整个 plan

3. **新建 `agent/core/observation.py`** — **核心新增**
   - `observe(state, client, prompt_builder)`: 整理工具返回为结构化观察
   - `build_observation_prompt()`: 让 LLM 分析工具返回，提取 findings + next_step_suggestion
   - Agent 与 Workflow 最大区别:
     - Workflow: 工具结果 → Reviewer（机械评审）
     - Agent: 工具结果 → Observation（智能观察）→ 重新思考
   - 输出结构: `{tool, args, result, status, findings, next_step_suggestion, is_sufficient}`
   - findings: 从工具结果中提取的关键发现
   - next_step_suggestion: 建议下一步操作
   - is_sufficient: 当前观察是否已足够回答目标

4. **重构 `agent/core/nodes.py`**
   - 移除内联的 reasoner_node / tool_node 实现
   - 改为从 reasoner.py / tool_runner.py / observation.py 导入
   - 保留 memory_update_node + critic_node + 条件边
   - critic 和 memory_update 的 trajectory 展示优先使用结构化 findings

5. **更新 `agent/core/state.py`**
   - observation 字段注释增强: 原始字段 + 观察节点增强字段

6. **更新 `app/services/graph_service.py`**
   - 图节点新增 observer（LangGraph 不允许节点名与 state key 同名，用 observer 而非 observation）
   - 图流程: reasoner → (tool → observer → memory_update | memory_update) → critic → (reasoner | END)
   - 事件流处理新增 observer 节点: 展示 findings + next_step_suggestion
   - trace 事件增加 findings 和 next_step_suggestion 字段

### 验证结果
- 图节点: `['__start__', 'reasoner', 'tool', 'observer', 'memory', 'critic']`
- 完整导入链正常

### 阶段5: Critic 独立 + 三问评判

**新建 `agent/core/critic.py`**
- `critique(state, client, prompt_builder)`: 独立评判函数
- `build_critic_prompt()`: 三问评判 prompt
  1. 正确性 — 回答是否有依据？
  2. 完整性 — 是否遗漏关键步骤？
  3. 下一步 — 是否需要继续探索？

- score: 0-10 整数，10=完全解决
- issues: 发现的问题列表
- suggestion: 下一步建议
