## 同步 LLM 调用阻塞事件循环修复
- **核心问题**：graph_service.py 使用同步 OpenAI 客户端，5 处 `client.chat.completions.create` 调用阻塞事件循环 2-10秒
  - 单用户可能不明显，多用户时一个 LLM 调用会阻塞所有其他请求
  - chat_router.py 的 `await asyncio.sleep(0.01)` hack 只在事件之间让出控制，LLM 调用期间完全阻塞
- **修复内容**：
  1. `from openai import OpenAI` → `from openai import AsyncOpenAI`，client 改为 AsyncOpenAI
  2. 3 个 graph 节点 `def → async def`：
     - planner_node：纯字典 pass-through，无 IO，仅签名改为 async
     - executor_node：`registry.execute_tool → await asyncio.to_thread(registry.execute_tool, ...)`（本地文件 IO 用线程池）
     - reviewer_node：`client.chat.completions.create → await client.chat.completions.create`
  3. `run_graph_stream` 从 sync generator → async generator：
     - 所有 `for chunk in client.chat.completions.create(stream=True)` → `async for chunk in await client.chat.completions.create(stream=True)`
     - `graph.stream()` → `graph.astream()`
     - `build_context` / `get_skill_context` / `list_capabilities` / `get_user_profile` / `get_pending_tasks` → 全部用 `asyncio.to_thread` 包装
     - `_build_planner_prompt` 新增可选 `user_profile` 和 `pending_info` 参数，run_graph_stream 预计算后传入（避免 prompt 构建函数内有同步调用）
     - `update_profile` / `save_pending_task` → `await asyncio.to_thread(...)` 包装
  4. `chat_service.py` `chat_with_agent_stream` → `async def`，内部 IO 用 `asyncio.to_thread` 包装
  5. `chat_router.py` `_stream_events` → `async for`，移除 `asyncio.sleep(0.01)` hack
- **效果**：LLM 调用不再阻塞事件循环，多用户并发时其他请求可正常处理
